# Serbito — Исследование рынка (Market Research)

**Проект:** Serbito — площадка заявок на доставку + сквозные уведомления (last-mile) в Сербии
**Дата:** 2026-05-31
**Метод:** 4 параллельных исследовательских потока (конкуренты, объём рынка, глобальные бенчмарки, боль + регуляторика)
**Аналитик:** Mary (BMAD Business Analyst)

> Условные обозначения силы доказательств: **[ТВЁРДО]** — именованный источник с цифрой; **[СРЕДНЕ]** — отраслевая/вторичная оценка; **[ОСТОРОЖНО]** — единичный отзыв/допущение.

---

## 1. Проблема подтверждена

**[ТВЁРДО — регуляторная статистика RATEL]**
- **53%** жалоб пользователей почтовых услуг — недоставка до двери квартиры/дома.
- **59%** — задержки доставки.
- За 2024: при падении объёма посылок на 2% рекламации выросли на **+6%**; 70.3% компенсаций — за утерянные посылки.
- Источник: RATEL «Prigovori korisnika poštanskih usluga», отчёт о качестве почтовых услуг 2024.

**[СРЕДНЕ — замер потребителей]**
- «Каждый пятый» (>20%) сталкивался с проблемой курьерской службы (b92.net, sd.rs).

**[ОСТОРОЖНО — единичные, но однотипные отзывы: Biznis.rs, Republika, Informer, Kurir, Trustpilot, форумы]**
- Курьер звонит уже у подъезда либо не звонит вовсе.
- Ложные «никого не было» при том, что получатель дома весь день.
- Курьеры не ждут / заставляют спускаться к подъезду; «мёртвый» трекинг (статус не обновляется).

**Центральные боли, на которые ложится продукт:**
1. Нет заблаговременного уведомления и нет интервала/ETA.
2. Ложные «никого не было» и возвраты.
3. Курьер не ждёт / конфликт по времени и месту.
4. «Мёртвый» трекинг.
5. COD-трение (нет сдачи, требование повторной оплаты).

## 2. Конкурентный ландшафт — дыра реальна

| Игрок | Трекинг | Проактивное уведомление + ETA получателю |
|---|---|---|
| Pošta Srbije / PostExpress | Да | Преим. реактивно (SMS при неудачной доставке); ETA — нет |
| D Express | Да | **Предупреждает в день доставки** (личный опыт владельца — лучший среди локальных); НО интервал/ETA — нет, live-трекинг — нет |
| BEX | Да | Не подтверждено; ETA — нет |
| AKS | Да | Платно по запросу (24 RSD); ETA — нет |
| City Express | Да | SMS по запросу (~12 RSD); ETA — нет |
| **GLS Serbia** | Да | **ДА: проактивно + 3-часовой интервал + тел. курьера** (контрпример — доказал, что возможно) |
| DHL Express | Да | Да (On Demand Delivery) — но премиум/международный |
| Glovo / Wolt | Live-карта | Да, live ETA — но это еда, не посылки |

**Вывод:** гипотеза подтверждается для массового внутреннего рынка. Локальные курьеры почти не дают «сегодня доставка + интервал». GLS — единственное исключение в parcel-сегменте → планка задана, конкуренты её не держат.

**Биржи грузов на Балканах** (BalkansCargo, Transportna Berza, CargoAgent, TIMOCOM RS): уведомляют только участников сделки об откликах, **получателя — никто**. Значимая часть рынка работает через хаотичные **Viber-группы** (BalkansCargo прямо атакует это в позиционировании).

## 3. Размер и привлекательность рынка

**[ТВЁРДО]**
- E-commerce Сербии: **~$0.85–1.1 млрд** (узкий B2C, 2024–2025), рост **+7–15%/год**.
- **82.4 млн** онлайн-покупок за 2024 (~225 тыс./день); доля онлайн-покупателей 34%→62% за 5 лет.
- Интернет **91.8%**, смартфоны **~97%**, **Viber у >90% населения** (доминирующий канал на Балканах).
- **COD ~70%** заказов (до ~85% по части источников) — структурная боль, доля даже росла +5 п.п. за 2019–2024.

**[СРЕДНЕ — отраслевые оценки]**
- Failed-first-attempt по COD: **30–50%** отказов у двери; 5–15% вовсе недоставлены.
- Оценка потерь: ~17 млн проблемных доставок/год × €2–4 = **~€35–70 млн/год** только в Сербии.

**Масштабирование:** Западные Балканы — e-commerce ~€3.3 млрд, >17 млн чел., та же COD/Viber-модель → TAM ×3–4 без смены продукта.

**Риски данных:** точный parcel-volume за пейволлом (Effigy/Statista) — дозакупить/запросить RATEL; демография падает (−0.7%/год); failed-delivery — отраслевые, не гос. цифры.

## 4. Глобальные бенчмарки (что заимствовать)

**Уведомления/трекинг:** AfterShip, parcelLab, Narvar (post-purchase для магазинов); Onfleet, Bringg, Track-POD, Detrack (для тех, кто возит); Route (трекинг бесплатно, монетизация через страховку).
**Биржи/матчинг:** Uber Freight/sennder (брокеры — не наша лига); Trans.eu/TIMOCOM (подписка, тренд к бесплатному входу); **Shiply/Clicktrans (reverse-auction — ближайший референс)**.

**Применимо к Serbito:**
1. **Viber №1**, SMS fallback, Email опц.
2. **Сквозные уведомления всем 3 ролям** (перевозчик/отправитель/получатель) — этого нет ни у одной локальной биржи.
3. **Публичная трекинг-страница по ссылке** (без приложения), white-label.
4. ETA + статус-чекпоинты (даже простые, без ML).
5. ePOD-лайт (фото/подпись/код).
6. Opt-in подписка получателя.

## 5. Регуляторика и юр. ограничения

- **Регулятор — RATEL.** Почтовая лицензия **вероятно НЕ нужна**, пока продукт только связывает и уведомляет (не принимает/возит/вручает физически). НО определение «почтовой услуги» включает «электронное отслеживание отправления» → **до запуска получить письменное мнение RATEL** по конкретной модели. *(средняя уверенность)*
- **Перс. данные — ZZPL** (GDPR-подобный, регулятор Poverenik):
  - Уведомления получателю слать на основании **исполнения договора** (НЕ consent).
  - Тело сообщения **строго сервисное** (иначе включается opt-in для маркетинга + Do-Not-Call регистр).
  - DPA-договор с SMS-шлюзом/облаком; breach-нотификация 72 ч.
  - SMS: зарегистрированный **alphanumeric sender ID** на MTS/Yettel/A1 (~16 дней пре-регистрации), часы 09:00–20:00, opt-out в теле.

## 6. Главный вывод: дифференциация Serbito

Serbito — **не «ещё одна доска заявок»**. Незанятая ниша = **сквозные уведомления получателю + простая публичная трекинг-страница + Viber-first**, поверх reverse-auction матчинга заявка↔перевозчик.

- **Модель матчинга:** заявка → отклики с ценой → выбор (как Shiply/Clicktrans/BalkansCargo).
- **Старт:** бесплатно для обеих сторон (набор ликвидности).
- **Монетизация позже:** премиум-подписка перевозчиков (~€10/мес, как BalkansCargo), либо допуслуги (страховка à la Route), либо платные SMS/Viber-пакеты (естественный апселл на уведомлениях).
- **Связь с кодом:** текущий `transport_site` (Request↔Response) — валидная основа, к ней добавляется notification-слой.
